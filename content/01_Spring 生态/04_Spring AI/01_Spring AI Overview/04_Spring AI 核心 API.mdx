# Spring AI 核心 API

Spring AI 试图用一套 Java 接口统一 AI 集成的各个环节。对于开发者来说，这意味着你的业务代码不会被某个特定厂商（比如 OpenAI）锁死——至少在通用能力的范围内是这样。

前面我们讲了核心概念，这一篇来看看这些概念在 Spring AI 里对应的 **API 长什么样、怎么用、以及用的时候要注意什么**。

Spring AI 的 API 版图虽然宏大，但核心围绕着**五根支柱**展开：

1. **AI Model API** — 对话、生成、嵌入等模型交互
2. **ChatClient API** — 面向业务的高阶统一入口
3. **Vector Store API** — 向量存储与检索（RAG 基石）
4. **Tool Calling API** — 让模型 "动手" 调用你的代码
5. **Advisors API** — AI 调用链上的中间件 / 拦截器

搞懂这五个，剩下的就是查文档的体力活了。

## 1. AI Model API（模型接口层）

如果说 `ChatClient` 是给驾驶员用的**方向盘和仪表盘**，那么 AI Model API 就是这辆车的**发动机和传动系统**。

虽然在日常业务开发中，我们 90% 的时间都在用 `ChatClient`，但理解 Model API 对于排查底层问题、自定义模型扩展以及理解 Spring AI 的抽象设计至关重要。

### 1.1 接口层次

Spring AI 的模型接口是一套泛型继承体系：

![](http://refienx-notes.oss-cn-shanghai.aliyuncs.com/blog/219wzb.png)

```
Model<TReq, TRes>              ← 所有模型的顶层接口
  ├── ChatModel                ← 对话模型（Prompt → ChatResponse）
  ├── StreamingChatModel       ← 流式对话模型（Prompt → Flux<ChatResponse>）
  ├── EmbeddingModel           ← 向量模型（文本 → float[]）
  ├── ImageModel               ← 图像生成模型
  ├── AudioTranscriptionModel  ← 语音转文字
  └── TextToSpeechModel        ← 文字转语音
```

**关键接口说明**：

- **`Model<TReq, TRes>`**：顶层泛型接口，定义了 "输入请求 → 返回响应" 的基本行为。你几乎不会直接用它，但理解它有助于理解整个继承体系。
- **`ChatModel`**：处理 `Prompt → ChatResponse` 的同步调用。这是你会在代码中看到的底层接口。
- **`StreamingChatModel`**：同样处理 `Prompt`，但返回 `Flux<ChatResponse>`——token 片段逐个推送，适合流式 UI。
- **`EmbeddingModel`**：把文本转成向量（`float[]`）。RAG 场景的必备组件。

### 1.2 核心接口解构：ChatModel

`ChatModel` 是一个同步（Blocking）接口，它的核心职责非常单一且纯粹：**将标准化的输入（Prompt）发送给 AI，并接收标准化的输出（ChatResponse）。**

无论后端接的是 OpenAI、DeepSeek、Azure 还是本地运行的 Ollama，对于上层应用来说，它们都是一个 `ChatModel`。这种设计完全遵循了 Spring 一贯的 "依赖倒置" 原则——**面向接口编程，而不是面向实现编程**。

让我们看看它的核心定义：

```java
public interface ChatModel extends Model<Prompt, ChatResponse> {

    // 1. 最核心的抽象方法：接收 Prompt，返回 ChatResponse
    @Override
    ChatResponse call(Prompt prompt);

    // 2. 快捷方法：只传字符串，适合简单测试
    default String call(String message) {
        Prompt prompt = new Prompt(new UserMessage(message));
        // 内部自动解包，提取文本内容
        Generation generation = call(prompt).getResult();
        return (generation != null) ? generation.getOutput().getText() : "";
    }
    
    // ... 其他重载方法
}
```

两个核心对象需要理解：

- **`Prompt`（输入）**：这是个容器，里面不仅装着用户的提问（`UserMessage`），还可以装系统设定（`SystemMessage`）和各种参数（`ChatOptions`，比如温度 Temperature）。
- **`ChatResponse`（输出）**：这是个大对象，里面包含了 AI 生成的内容（`Generation`）、元数据（`Metadata`）以及 Token 消耗统计（`Usage`）。

这套抽象带来的好处很直接：

- **屏蔽差异**：OpenAI 的 API 叫 `messages`，Google 的叫 `contents`，但在 Spring AI 里，它们统统被封装成了 `Prompt`。
- **无缝切换**：今天用 `OpenAiChatModel`，明天想换成 `OllamaChatModel`，业务代码里的 `call()` 方法完全不需要动，只需要改改配置。

虽然我们推荐用 `ChatClient`，但直接注入 `ChatModel` 也是完全合法的，特别是在写一些基础工具库时：

```java
@RestController
public class DirectModelController {

    private final ChatModel chatModel;

    // Spring Boot 自动注入配置好的实现类（如 OpenAiChatModel）
    public DirectModelController(ChatModel chatModel) {
        this.chatModel = chatModel;
    }

    @GetMapping("/raw-chat")
    public String simpleChat(String msg) {
        // 直接调用底层 API，没有 ChatClient 那些花哨的封装
        return chatModel.call(msg);
    }
}
```

### 1.3 流式接口：StreamingChatModel

在 AI 应用中，"速度感" 是用户体验的关键。让用户盯着空白屏幕等 5 秒钟是不可接受的。`StreamingChatModel` 就是为了解决这个问题，利用 Java 的 Reactive Stack（Reactor）实现流式响应。

接口定义如下：

```java
@FunctionalInterface
public interface StreamingChatModel extends StreamingModel<Prompt, ChatResponse> {
    
    // 核心方法：返回 Flux 流
    @Override
    Flux<ChatResponse> stream(Prompt prompt);

    // 快捷方法：直接返回字符串流
    default Flux<String> stream(String message) {
        // ... 自动封装 Prompt 并提取文本流
    }
}
```

<Callout type="info" title="为什么是 Flux？">
  1. `Flux` 是 Project Reactor 的核心类，代表一个包含 0 到 N 个元素的异步序列。  
  2. 大模型每生成一个字（或几个 Token），就会向这个流里 `emit` 一个事件。前端通过 SSE（Server-Sent Events）订阅这个流，就能实现 ChatGPT 那种“打字机”效果。
</Callout>

### 1.4 ChatModel 与各大厂商的关系

Spring AI 通过适配器模式（Adapter Pattern），将各大厂商的 SDK 统一到了 `ChatModel` 之下。

主要实现类一览：

| **实现类名称** | **对应厂商 / 技术** | **特点** |
| --- | --- | --- |
| `OpenAiChatModel` | OpenAI | 最标准、功能最全，支持 ChatGPT 全系列 |
| `OllamaChatModel` | Ollama | **本地党最爱**，支持 Llama 3、Qwen、Mistral 等本地模型 |
| `AzureOpenAiChatModel` | Microsoft Azure | 企业级 OpenAI 服务，安全性更高 |
| `BedrockChatModel` | AWS Bedrock | 亚马逊云用户的首选 |
| `VertexAiGeminiChatModel` | Google | 支持 Google 最新的 Gemini 模型 |

Chat Model API 支持的 AI 模型厂商如下：

![](http://refienx-notes.oss-cn-shanghai.aliyuncs.com/blog/xm3v61.png)

## 2. ChatClient API（业务层核心入口）

如果说 `ChatModel` 是连接 AI 的 "底层驱动"，那么 `ChatClient` 就是 Spring AI 精心为你打造的 "高级驾驶舱"。它把繁琐的底层交互封装成了优雅的流式 API，让你写 AI 代码像写 Stream 流一样顺滑。

### 2.1 为什么需要 ChatClient

你可能会问："既然有了 `ChatModel`，直接调用不行吗？"

可以，但不够优雅。

`ChatModel` 的职责是 **"标准化底层调用"**，它的 API 往往比较原始。而 `ChatClient` 的职责是 **"提升开发体验"**。`ChatClient` 是一个接口，默认只有一个 `DefaultChatClient` 实现类。

![](http://refienx-notes.oss-cn-shanghai.aliyuncs.com/blog/m5yhub.png)

我们可以用一个经典的 Java 数据库操作做类比：

- **ChatModel ≈ JDBC Driver**：它是底层的连接器，不同的厂商（OpenAI、DeepSeek）有不同的实现。
- **ChatClient ≈ JdbcClient / JdbcTemplate**：它封装了复杂的底层逻辑，提供了 Fluent API（流式编程）、Advisor（拦截器）等高级特性。

简单来说：**`ChatModel` 是给机器用的，`ChatClient` 才是给你用的。**

### 2.2 核心入口：`prompt()` 的三种姿势

`ChatClient` 的核心是 `prompt()` 方法，它是一切对话的起点。它有三个重载版本，满足不同场景的需求：

<Tabs>
  <Tab title="零参数（推荐）">
    这是最灵活的方式，允许你通过链式调用一步步构建复杂请求：

    ```java
    String response = chatClient.prompt()
        .system("你是一个专业的 Java 助手")   // 设定人设
        .user("Spring AI 的核心优势是什么？") // 用户提问
        .call()     // 发起同步调用
        .content(); // 获取文本内容
    ```
  </Tab>
  <Tab title="传字符串">
    如果你只需要简单问一句话，不需要系统预设：

    ```java
    String response = chatClient.prompt("讲个笑话").call().content();
    ```
  </Tab>
  <Tab title="传 Prompt 对象">
    如果你已经在别处构建好了 `Prompt` 对象（可能包含复杂 Message 列表），可以直接传入：

    ```java
    Prompt myPrompt = new Prompt(...);
    chatClient.prompt(myPrompt).call().content();
    ```
  </Tab>
</Tabs>

### 2.3 处理返回结果

AI 返回的数据不仅是文本，还包含 Token 消耗、元数据等重要信息。`ChatClient` 提供了多种方式来处理返回结果。

<Tabs>
  <Tab title=".content() 文本">
    最简单的方式，直接拿到模型输出的纯文本。前面的示例已经反复用到了。
  </Tab>
  <Tab title=".chatResponse() 细节">
    如果你需要知道这顿操作消耗了多少 Token（毕竟这直接关联到**钱**），或者需要查看 AI 的原始元数据：

    ```java
    ChatResponse response = chatClient.prompt("分析财报").call().chatResponse();

    // 获取 Token 使用情况（计费关键！）
    Usage usage = response.getMetadata().getUsage();
    System.out.println("消耗 Token: " + usage.getTotalTokens());
    ```
  </Tab>
  <Tab title=".entity() 对象">
    这是 Spring AI 最强大的功能之一，自动将 AI 的自然语言回复解析为 Java Bean：

    ```java
    // 定义接收对象（Record 是绝配）
    record UserAddress(String username, List<String> addressList) {}

    // 调用
    UserAddress result = chatClient.prompt()
        .user("帮我把'张三住在北京朝阳，李四在上海浦东'整理一下")
        .call()
        .entity(UserAddress.class); // 自动反序列化

    System.out.println(result.username()); // 输出：张三
    ```

    如果返回的是列表，可以使用 `ParameterizedTypeReference`：

    ```java
    List<UserAddress> users = chatClient.prompt("...")
        .call()
        .entity(new ParameterizedTypeReference<List<UserAddress>>() {});
    ```

    Spring AI 会在底层自动做两件事：
    1. 在 Prompt 末尾附加 JSON Schema 指令，引导模型按结构输出。
    2. 将模型返回的 JSON 反序列化为你的 Java Bean。
  </Tab>
  <Tab title=".stream() 流式">
    对于长文本生成，不要让用户等太久。使用 `.stream()` 配合 `Flux`，实现像 ChatGPT 那样“打字机”的效果：

    ```java
    Flux<String> stream = chatClient.prompt("写一篇 500 字的小说")
        .stream() // 切换为流式模式
        .content();

    // 前端通过 SSE (Server-Sent Events) 订阅这个 Flux 即可
    ```
  </Tab>
</Tabs>

### 2.4 全局默认配置与动态参数

在实际项目中，我们不希望每次都写 `system("你是一个...")`。`ChatClient` 支持全局默认配置。

**全局配置**

你可以在 Spring 配置类中，预先组装好一个带有 "默认人设" 的 Client：

```java
@Configuration
class AiConfig {
    @Bean
    ChatClient myChatClient(ChatClient.Builder builder) {
        return builder
            // 默认系统人设
            .defaultSystem("你是一个得力的翻译助手，请用 {style} 的风格翻译。")
            // 默认 Advisor（用于日志追踪）
            .defaultAdvisors(new SimpleLoggerAdvisor()) 
            .build();
    }
}
```

**动态参数替换**

上面的 `{style}` 是一个占位符。在运行时，我们不需要重写 System Text，只需要传入参数：

```java
chatClient.prompt()
    .system(s -> s.param("style", "莎士比亚")) // 动态替换
    .user("Hello World")
    .call()
    .content();
```

### 2.5 消息元数据（Metadata）

你可以给消息打上标签（Tag），这些标签不会发给 AI 模型，但会贯穿整个调用链路，方便在日志或拦截器中追踪：

```java
.user(u -> u.text("...")
    .metadata("userId", "1001") // 追踪是谁调用的
    .metadata("bizId", "order-123"))
```

也可以通过 `Map<String, Object>` 一次性传入：

```java
Map<String, Object> metadata = new HashMap<>();
metadata.put("userId", "1001");
metadata.put("bizId", "order-123");

.user(u -> u.text("...").metadata(metadata))
```

同理，系统消息也同样支持元数据注入，方式和用户消息没什么区别。

<Callout type="note" title="Metadata 校验规则">
  ChatClient 会校验元数据，确保数据完整性和可靠性：
  1. 元数据的键（key）不能为 `null` 或空字符串；  
  2. 元数据的值（value）不能为 `null`，可以是空字符串；  
  3. 传 `Map` 的时候，key 和 value 都不能为 `null`。
</Callout>

上面添加的元数据会自动包含在生成的 `UserMessage` 和 `SystemMessage` 对象里，你可以通过消息的 `getMetadata()` 方法轻松调用，这在处理 Advisor 消息或者查看对话历史时特别实用。

### 2.6 带工具调用

```java
String response = chatClient
    .prompt("查一下商品 P001 的库存")
    .tools(new StockService())   // 注册工具
    .call()
    .content();
```

Spring AI 会自动处理 "模型请求调用工具 → 执行 Java 方法 → 结果回传给模型 → 生成最终回答" 这整个循环。（详见[第 4 节 Tool Calling API](https://www.notion.so/Spring-AI-API-30c4cb8807f280929dd6f4ffd51ecfa5?pvs=21)）

### 2.7 带 Advisor

```java
ChatClient chatClient = ChatClient.builder(chatModel)
    .defaultAdvisors(
        new MessageChatMemoryAdvisor(chatMemory),   // 自动注入对话历史
        new QuestionAnswerAdvisor(vectorStore)      // 自动做 RAG 检索
    )
    .build();
```

配置好 Advisor 之后，每次调用 `chatClient.prompt(...).call()` 时，Advisor 链会自动执行——你不需要在业务代码里手动拼历史消息或检索文档。（详见[第 5 节 Advisors API](https://www.notion.so/Spring-AI-API-30c4cb8807f280929dd6f4ffd51ecfa5?pvs=21)）

### 2.8 Chat Memory（对话记忆）

默认情况下，LLM 是**无状态**的。你问它 "我叫什么？"，它回答后就忘了。下一句你问 "我刚才说了什么？"，它就懵了。为了实现连续对话，我们需要 **Chat Memory**。

**工作原理**

`ChatClient` 通过 **Advisor** 机制集成记忆功能。它会在发请求前，自动把历史聊天记录从存储中取出来，拼接到 Prompt 里。

**核心组件：MessageWindowChatMemory**

ChatMemory 目前内置了一个最常用的实现 `MessageWindowChatMemory`，基于 "滑动窗口" 机制：

- **原理**：只保留最近的 N 条消息（默认 20 条）。
- **优势**：防止 Token 超限，同时节省成本。
- **特性**：System Message（系统人设）永远不会被删除，只删普通的 User/Assistant 对话。

![](http://refienx-notes.oss-cn-shanghai.aliyuncs.com/blog/1uyuw6.png)

**存储介质**

Spring AI 提供了多种存储实现（`ChatMemoryRepository`）：

- **`InMemoryChatMemoryRepository`**：内存存储，重启就丢（适合测试）。
- **`JdbcChatMemoryRepository`** / **`CassandraChatMemoryRepository`** / **`Neo4jChatMemoryRepository`**：持久化存储（生产环境必备）。

**实战配置示例**：

```java
@Bean
ChatClient chatClient(ChatClient.Builder builder, ChatMemory chatMemory) {
    return builder
        .defaultAdvisors(new MessageChatMemoryAdvisor(chatMemory)) // 启用记忆增强
        .build();
}
```

### 2.9 ChatModel vs ChatClient：什么时候用哪个

这是初学者最容易混淆的问题。来一张全面对比：

| **维度** | **ChatModel** | **ChatClient** |
| --- | --- | --- |
| 定位 | 基础设施层（Driver） | 应用层（Builder） |
| API 风格 | 传统方法调用 | 链式 Fluent API（类似 WebClient） |
| Advisor 支持 | 不支持 | 内置 Advisor 链（拦截器机制） |
| 流式支持 | 需要单独用 StreamingChatModel | `.call()` 同步 / `.stream()` 流式，一个入口搞定 |
| 结构化输出 | 需要手动处理 | 内置 `.entity()` 直接映射 POJO |
| 记忆管理 | 不支持 | 通过 Advisor 集成 ChatMemory |
| 可测试性 | Mock 简单，适合单元测试 | Mock 相对复杂 |
| 推荐场景 | 封装底层组件、自定义框架、访问冷门底层参数 | **业务代码首选**（99% 的情况用它） |

<KeyPoint title="ChatClient 设计哲学">
  `ChatClient` 的目标是把工具调用、RAG、记忆、格式化等复杂性收敛到一个统一入口，业务代码只需要关注“问什么”和“拿到什么”。
</KeyPoint>

## 3. Vector Store API（向量存储 API）

如果说 Model API 是 AI 的大脑，那么 Vector Store API 就是 AI 的**长期记忆**。它是实现 RAG 架构的基石。

### 3.1 VectorStore 接口

市面上的向量数据库五花八门——Redis、Pinecone、Milvus、Qdrant、PGVector、Chroma、Weaviate... 每家的查询语法都不一样。Spring AI 将它们统一封装成 `VectorStore` 接口，核心方法就三个：

```java
public interface VectorStore {
    // 写入：把文档（已含 Embedding 向量）存进去
    void add(List<Document> documents);

    // 查询：根据语义相似度检索
    List<Document> similaritySearch(SearchRequest request);

    // 删除：按 ID 移除
    void delete(List<String> ids);
}
```

你只需要面向这个接口编程，底层换数据库时业务代码不需要改——和 Spring Data 的 Repository 抽象是同一个思路。

### 3.2 元数据过滤（Metadata Filtering）

这可能是 Spring AI 在 Vector Store 层最实用的功能之一。在实际业务中，我们不仅要 "找语义相似的"，还经常要 "按条件过滤"。比如：

<QuoteCard source="检索需求示例">
  找和 "Java 内存模型" 相关的文档，但必须是 2026 年发布的，且作者是张三。
</QuoteCard>

如果只靠向量相似度，做不到这种精确筛选。Spring AI 提供了一套**类似 SQL 的元数据过滤 API**，并且它是跨数据库可移植的：

```java
FilterExpressionBuilder b = new FilterExpressionBuilder();

List<Document> results = vectorStore.similaritySearch(
    SearchRequest.query("Java 内存模型")
        .withTopK(5) // 返回数量
        .withSimilarityThreshold(0.7) // 最低相似度
        .withFilterExpression( // 元数据过滤
            b.and(
                b.eq("author", "张三"),
                b.gte("year", 2026)
            ).build()
        )
);
```

这段代码无论底层用的是 Redis、PostgreSQL（PGVector）还是 Milvus，Spring AI 都会自动把过滤表达式翻译成对应数据库的查询语句，这一点不得不给 Spring 团队点赞 👍。

### 3.3 工程建议

<Checklist title="Vector Store 工程建议">
  <CheckItem checked>
    **存文档时一定要带上元数据**（作者、日期、来源、类别、租户 ID 等）。元数据是后续精确检索的前提，不带就只能“模糊找”。
  </CheckItem>
  <CheckItem checked>
    **`topK` 和 `similarityThreshold` 要配合调**：`topK` 控制返回数量，`similarityThreshold` 控制最低相似度。两个参数直接影响检索质量和成本。
  </CheckItem>
  <CheckItem checked>
    **向量维度由 Embedding 模型决定**，不同模型产出的维度不同（比如 OpenAI 的 `text-embedding-3-small` 是 1536 维）。换 Embedding 模型意味着需要重建索引。
  </CheckItem>
</Checklist>

## 4. Tool Calling API（工具调用 API）

这是 AI 从 "聊天机器人" 进化为 "智能助理（Agent）" 的关键能力。

大模型本身是封闭的——它不知道现在的天气，也没法帮你查数据库。Tool Calling（OpenAI 称之为 Function Calling）允许你把 Java 代码 "注册" 给模型，让模型在需要时主动调用。

![](http://refienx-notes.oss-cn-shanghai.aliyuncs.com/blog/benjuk.png)

### 4.1 工作流程

<Steps>
  <Step title="用户发起问题">
    用户提问：“查一下商品 P001 的库存”。
  </Step>
  <Step title="模型决策调用工具">
    模型分析这个问题需要外部数据，决定调用 `getStock` 工具。
  </Step>
  <Step title="返回工具调用请求">
    模型返回请求：`{ tool: "getStock", args: { productId: "P001" } }`。
  </Step>
  <Step title="Spring AI 执行本地方法">
    Spring AI 自动拦截并执行你的 Java 方法：`getStock("P001")`。
  </Step>
  <Step title="工具结果回传模型">
    工具返回库存值（例如 `100`），Spring AI 再把结果喂回模型。
  </Step>
  <Step title="模型生成最终回答">
    模型组织自然语言回复：“商品 P001 当前库存为 100 件。”
  </Step>
</Steps>

整个过程中，**模型本身不执行任何代码**——它只是 "决定" 调哪个工具、传什么参数。实际执行发生在你的 Spring 应用里。

### 4.2 写法一：@Tool 注解（推荐）

最符合 Spring 风格的写法，直接在方法上加注解：

```java
@Service // 需要是 Spring Bean
public class StockService {

    @Tool(description = "根据商品ID查询当前库存数量") // 描述并注册为 Tool
    public int getStock(String productId) {
        // 查询数据库...
        return inventoryRepository.countByProductId(productId);
    }
}
```

然后在 ChatClient 调用时注册这个工具：

```java
String response = chatClient
    .prompt("查一下商品 P001 的库存")
    .tools(stockService) // 注册工具，会找到其中使用了 @Tool 的方法作为工具注册
    .call()
    .content();
```

Spring AI 会自动把方法签名、参数类型、`description` 转换成模型能理解的 Tool Definition，并在模型触发调用时自动执行。

### 4.3 写法二：Java Function Bean

适合将第三方库的方法或不方便加注解的逻辑包装成工具：

```java
@Bean
@Description("获取某城市的实时天气信息")
public Function<WeatherRequest, WeatherResponse> weatherFunction() {
    return request -> weatherService.query(request.city());
}
```

### 4.4 工程要点与安全边界

<Callout type="warning" title="Tool Calling 安全边界">
  Tool Calling 赋予了模型“调用你代码”的能力，这意味着安全性需要你自己把控：
  - **最小权限原则**：不要把删除、转账等危险操作注册为工具，除非你有完善的确认机制。  
  - **输入校验**：模型传过来的参数不一定合法（比如注入恶意字符串），你的工具方法必须做参数校验，和对待外部用户输入一样严格。  
  - **工具描述质量会直接影响调用准确率**：`description` 写得越清晰，模型越能准确判断什么时候该调用、传什么参数。模糊描述会导致误调或漏调。  
  - **模型不一定会调用工具**：即使你注册了工具，模型也可能判断“不需要调用”而直接回答，这是正常行为，不是 bug。
</Callout>

## 5. Advisors API（AI 中间件）

Advisors 是 Spring AI 中一个很有 "Spring 味道" 的设计——如果你用过 Spring MVC 的 `HandlerInterceptor` 或 Spring AOP 的切面，会立刻找到熟悉的感觉。

### 5.1 Advisor 做什么

Advisor 是 ChatClient 调用链上的**拦截器**，可以在请求发送给模型**之前**和模型返回结果**之后**做处理：

<Steps>
  <Step title="请求进入调用链">
    用户调用 `ChatClient.prompt().call()`。
  </Step>
  <Step title="前置增强 1">
    注入对话历史（Chat Memory）。
  </Step>
  <Step title="前置增强 2">
    检索相关文档并拼入 Prompt（RAG）。
  </Step>
  <Step title="前置增强 3">
    添加结构化输出或风格约束等指令。
  </Step>
  <Step title="模型执行与后置处理">
    请求发送给模型，随后执行后置逻辑：解析输出、记录检索命中、保存本轮对话。
  </Step>
  <Step title="响应返回">
    最终结果返回给调用方。
  </Step>
</Steps>

### 5.2 Spring AI 内置的常用 Advisor

| **Advisor** | **作用** | **典型场景** |
| --- | --- | --- |
| `MessageChatMemoryAdvisor` | 自动注入对话历史 + 保存本轮对话 | 多轮对话 |
| `QuestionAnswerAdvisor` | 自动检索 VectorStore 并把结果拼入 Prompt | RAG / 知识库问答 |
| `SafeGuardAdvisor` | 对输入/输出做安全过滤 | 内容安全、敏感词过滤 |

### 5.3 自定义 Advisor

你也可以实现自己的 Advisor 来封装业务逻辑——比如：

- 在每次请求前注入用户的权限信息
- 在每次响应后记录审计日志
- 在 prompt 中动态添加当前时间、用户身份等上下文

```java
public class AuditAdvisor implements CallAroundAdvisor {
    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
        // 前置：记录请求
        log.info("AI 请求: {}", request.userText());

        // 执行调用链
        AdvisedResponse response = chain.nextAroundCall(request);

        // 后置：记录响应
        log.info("AI 响应: {}", response.response().getResult().getOutput().getText());

        return response;
    }
}
```

<KeyPoint title="Advisors 的核心价值">
  Advisors 把记忆、检索、安全、审计等横切关注点从业务代码中剥离出来，形成可组合、可复用、可独立测试的组件。这与 Spring 生态一贯的设计哲学一致。
</KeyPoint>

## 6. 五大 API 的协作关系

这五个 API 不是孤立存在的，它们在一个典型的 AI 应用中会这样协作：

<Steps>
  <Step title="统一入口接收请求">
    用户问题先进入 `ChatClient`。
  </Step>
  <Step title="Advisor 前置处理">
    常见前置逻辑包括：`ChatMemoryAdvisor` 注入历史、`QuestionAnswerAdvisor` 检索知识库。
  </Step>
  <Step title="组装最终 Prompt">
    合并 System 指令、历史消息、检索结果和当前用户问题。
  </Step>
  <Step title="调用 ChatModel">
    通过 Model API 把请求发送到具体模型提供商。
  </Step>
  <Step title="按需执行 Tool Calling">
    如果模型判断需要工具调用，就触发 Tool Calling API 执行 Java 方法并把结果回传模型。
  </Step>
  <Step title="Advisor 后置处理与返回">
    保存记忆、记录审计日志，最终把结果返回给用户。
  </Step>
</Steps>

## 7. 总结

Spring AI 的五个核心 API 构成了现代 AI 应用的完整骨架：

<Checklist title="五大 API 快速回顾">
  <CheckItem checked>**Model API**：底层驱动，负责与各家模型通信（发动机）。</CheckItem>
  <CheckItem checked>**ChatClient API**：业务入口，Fluent API 统一同步/流式/结构化/工具调用/记忆（驾驶舱）。</CheckItem>
  <CheckItem checked>**Vector Store API**：向量存储与检索，RAG 的基石（长期记忆）。</CheckItem>
  <CheckItem checked>**Tool Calling API**：让模型调用你的代码，从“聊天”进化为“行动”（手脚）。</CheckItem>
  <CheckItem checked>**Advisors API**：AI 调用链上的中间件，封装横切关注点（神经系统）。</CheckItem>
</Checklist>

`ChatModel` 是 Spring AI 的基石——无论底层的 AI 模型如何风云变幻，上层的 Java 代码都应岿然不动。而 `ChatClient` 是你日常写代码时的主力武器——它用 Fluent API 解决了构建请求的繁琐，用 Entity Converter 解决了结构化输出的难题，用 Advisors + ChatMemory 解决了大模型无状态的痛点。

掌握了这五个 API 的定位和协作方式，你就具备了设计复杂 AI Agent 架构的基础。下一篇我们会深入实战，看看如何用这些 API 结成当下热门的服务提供商。
